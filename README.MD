## Functional Requirements

### Authentication

- **FR-AUTH-1:** A new user can register with a unique username, email address, and password.
- **FR-AUTH-2:** A registered user can log in with their email and password.
- **FR-AUTH-3:** A logged-in user receives a JWT token used to authenticate subsequent requests.
- **FR-AUTH-4:** A logged-in user can log out, clearing their session token.
- **FR-AUTH-5:** Protected routes and API endpoints are inaccessible without a valid token.

### Book Management

- **FR-BOOK-1:** An authenticated user can add a book to their shelf with the following fields:
  - Title (required)
  - Author (required)
  - Status: `reading` | `read` | `want_to_read` (required)
  - Rating: 1â€“5 (optional, only valid when status is `read`)
  - Notes: free text (optional)
- **FR-BOOK-2:** An authenticated user can view all books on their shelf.
- **FR-BOOK-3:** An authenticated user can view a single book's detail page.
- **FR-BOOK-4:** An authenticated user can edit any field of a book they own.
- **FR-BOOK-5:** An authenticated user can delete a book from their shelf.
- **FR-BOOK-6:** A user can only view and manage their own books â€” not other users' shelves.

### Filtering & Sorting

- **FR-FILTER-1:** A user can filter their shelf by status (`reading`, `read`, `want_to_read`).
- **FR-FILTER-2:** A user can sort their shelf by date added (newest/oldest) or by title (Aâ€“Z).

## Non-Functional Requirements

- **NFR-1 â€” Security:** Passwords must be hashed with `bcrypt` before being stored. Plain text passwords must never be persisted.
- **NFR-2 â€” Validation:** All incoming API request bodies must be validated with Zod before reaching business logic. Invalid requests return a `400` with a descriptive error message.
- **NFR-3 â€” Auth middleware:** JWT verification must be handled in a dedicated middleware function applied to all protected routes â€” not inline in route handlers.
- **NFR-4 â€” Error handling:** The backend must have a centralised error-handling middleware that catches unhandled errors and returns consistent JSON error responses.
- **NFR-5 â€” Environment config:** All secrets (JWT secret, DB connection string) must be managed via environment variables. No secrets in source code.
- **NFR-6 â€” Database migrations:** All schema changes must be managed via `node-pg-migrate` migration files. The database must never be modified by hand.
- **NFR-7 â€” Test coverage:** Critical paths (auth flow, book CRUD) must have integration test coverage on the backend and E2E test coverage via Playwright.
- **NFR-8 â€” Documentation:** The README must be kept up to date with setup instructions, environment variable reference, and how to run tests.

## API Requirements

Base path: `/api`

All protected endpoints require an `Authorization: Bearer <token>` header.

### Auth

| Method | Endpoint             | Auth   | Description         |
| ------ | -------------------- | ------ | ------------------- |
| `POST` | `/api/auth/register` | Public | Register a new user |
| `POST` | `/api/auth/login`    | Public | Log in, receive JWT |

### Books

| Method   | Endpoint         | Auth      | Description                    |
| -------- | ---------------- | --------- | ------------------------------ |
| `GET`    | `/api/books`     | Protected | Get all books for current user |
| `GET`    | `/api/books/:id` | Protected | Get a single book by ID        |
| `POST`   | `/api/books`     | Protected | Add a new book                 |
| `PUT`    | `/api/books/:id` | Protected | Update a book                  |
| `DELETE` | `/api/books/:id` | Protected | Delete a book                  |

The following slices must be managed in Redux global store:

- **`authSlice`** â€” current user, token, login/logout actions

### Backend (Node test runner + Supertest)

- Register endpoint â€” valid input creates a user; duplicate email returns 400; missing fields return 400
- Login endpoint â€” correct credentials return a token; wrong password returns 401
- Books CRUD â€” all endpoints return 401 without a token; CRUD operations work correctly with a valid token; a user cannot access another user's books

### Frontend (React Testing Library)

- Empty state renders when shelf is empty

### End-to-End (Playwright)

## Out of Scope

The following are explicitly excluded from v1 to keep scope manageable:

- Social features (following other users, public shelves)
- Book cover images or external book search APIs (e.g. Google Books)
- Reading progress tracking (page numbers, % complete)
- Email verification or password reset flow
- Mobile app
- Deployment / CI pipeline (may be added as a stretch goal)
